Lesson 0002 · Env vars in deplacementapp

Environment variables: where they live, where they go

Mission: deploy on Coolify · 6–8 min

You've already seen that a container gets its own copy of environment variables — inherited from the image, then overridden by Compose, then overridden by Coolify. But which env vars does deplacementapp actually need, and why does the wrong one crash it?

Three groups, one app

Open your repo's compose.yml and look at the app service's environment: block (lines 7–16). Every env var your app reads is listed there. They split into three groups:

GroupVariablesRequired?
Database DATABASE_URL Yes — no DB, no app
Auth NEXTAUTH_SECRET, NEXTAUTH_URL, AUTH_TRUST_HOST Yes — no auth, no login
Email SMTP_HOST, SMTP_PORT, SMTP_USER, SMTP_PASS, SMTP_FROM No — app runs, but notifications don't send

Where does the app read them?

Each env var is read by process.env.<NAME> somewhere in the code:

lib/prisma.ts const pool = new Pool({ connectionString: process.env.DATABASE_URL })
lib/email-service.ts const host = process.env.SMTP_HOST const port = process.env.SMTP_PORT const user = process.env.SMTP_USER const pass = process.env.SMTP_PASS const from = process.env.SMTP_FROM
prisma.config.js url: process.env.DATABASE_URL

NextAuth.js reads NEXTAUTH_SECRET, NEXTAUTH_URL, and AUTH_TRUST_HOST from env automatically — you don't see them in your own code, but the library pulls them on startup.

The production puzzle: localhost kills the container

Compare your three files:

FileDATABASE_URL hostWorks where
.env (local dev) localhost On your machine, because Postgres runs on your localhost
.env.example db Inside Docker Compose, because the Postgres container is named db
compose.yml db Inside Docker Compose (same reason)
Inside a Docker container, localhost means the container itself — not your host machine, not another container. So when the app container tries localhost:5432, it's knocking on its own door. Postgres isn't there. The container crashes with P1001.

This is why Coolify deployments fail: the DATABASE_URL you set in Coolify's env panel still says localhost from your dev file. The fix is simple: replace localhost with the hostname that reaches Postgres — the Docker container name (db) if Postgres is another container, or Coolify's internal database hostname if you're using Coolify's managed Postgres service.

What about .env itself?

The .env file is for your local machine. When you run docker compose up, Compose reads the .env file in the same directory and substitutes ${VAR} placeholders in compose.yml.

But inside the Docker container, the app doesn't read .env. The env vars reach the container through the environment: block in compose.yml (or via Coolify's env panel, or docker run -e). The container never touches your .env file.

Key insight

There are three separate mechanisms for setting env vars, and they don't mix:

  1. .env — local dev, read by Compose for substitution, not by the running app
  2. compose.yml environment: — what the container actually receives
  3. Coolify env panel — production env vars, injected at deploy time

Check your understanding

Your win for this lesson

You can now look at a DATABASE_URL that says localhost and know immediately: "that will crash inside a container." And you know which three groups exist, which are required, and how env vars actually reach the running app.

Do this next

  • Open your repo's .env and .env.example side by side. Notice which variables are in one but not the other — and why .env.example uses db while .env uses localhost.
  • Open Coolify → your app → Environment Variables. Check what DATABASE_URL currently says. If it says localhost, you've found the bug.
  • Primary source: Docker Compose environment variables — official docs on how env vars flow from .env → Compose → container.
