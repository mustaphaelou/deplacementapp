Lesson: Should you keep the query?

You're collapsing three guard layers into one. The question is: does canTransition survive?

The current design (before refactor)

demande-workflow.ts:

  // Layer 1: cheap boolean check (query)
  if (!canTransition(role, etape, action, decision))
    throw new UnauthorizedActionError()

  // Layer 2: build (command) — returns null on failure
  const result = buildTransition(role, etape, action, params)
  if (!result) throw new InvalidTransitionError()

Two functions, two error types, two checks. The handoff says: make buildTransition the single decider that throws instead of returning null.

Key insight: In the new design, buildTransition does double duty — it BOTH validates the transition AND builds the payload. That makes it a command that can throw.

The trade-off

If you keep canTransition as… Pro Con
Exported (public) UI can call it to show/hide action buttons without try/catch Two public entry points to reason about; someone might add a third layer again
Internal only (not exported) Still available inside workflow.ts for getAllowedActions; not leaked to callers None — this is the sweet spot if getAllowedActions has a future use
Deleted Zero dead code; one entry point No cheap query exists. If the UI needs action buttons later, you reintroduce a query or use try/catch

The crux: getAllowedActions

getAllowedActions currently has zero production callers — only tests. It wraps 4 calls to canTransition plus ownership checks:

getAllowedActions(role, userId, demande) → {
  canSubmit:   canTransition(r, etape, "submit",    decision) && isOwner,
  canApprove:  canTransition(r, etape, "approuver", decision),
  canReject:   canTransition(r, etape, "rejeter",   decision),
  canWithdraw: canTransition(r, etape, "retirer",   decision) && isOwner,
}
Warning: Even though getAllowedActions has no callers today, the reason it exists is obvious — the UI needs to know which buttons to render. Deleting it means work later when someone builds the UI actions panel.

The recommendation

The handoff suggests option keep it internal — and I agree. Here's why:

  • canTransition stays in workflow.ts but isn't exported from the module.
  • getAllowedActions still uses it internally — you delete getAllowedActions from exports too (zero callers), but the pattern is preserved inside the module if you ever need to re-export it.
  • buildTransition throws UnauthorizedActionError for role/policy guards and InvalidTransitionError for terminal/invalid state.
  • demande-workflow.ts drops the canTransition call — it just calls buildTransition and lets it throw.
