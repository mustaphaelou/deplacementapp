Image vs container — with a chainsaw

You're here because the first pass didn't land. That's fine; retrieval is hard on purpose. 5 min.

Last lesson you got the four nouns and a quiz. The quiz didn't go well — so forget the nouns for a minute. Let's make the distinction physically obvious by doing something bad to a running container and watching it not stick.

A recipe, then a cake

An image is a recipe. You build it once. It sits on a shelf, immutable. A container is a cake you baked from that recipe — it exists in time, you can poke it, and if you throw it away you can bake an identical one from the same recipe.

on the shelf
Image (recipe)
  1. built by docker build
  2. read-only, never changes
  3. identified by a hash (abc123…)
  4. does not answer a port
on the counter
Container (cake)
  1. created by docker run
  2. running, has a writable layer
  3. has a name like app-1
  4. the thing answering port 3000

The thing answering localhost:3000 after docker compose up is a container. The image is sitting on the shelf the whole time. Compose just happens to do build → run for you, so the two blur into one command.

Watch a container edit vanish

Here's the demo that makes the distinction concrete. Imagine you do this:

shell — inside a running container named app-1 docker exec app-1 sh -c 'echo "hello" > /tmp/marker' docker exec app-1 cat /tmp/marker # → hello docker stop app-1 && docker rm app-1 docker run ... # brand new container docker exec cat /tmp/marker # → No such file
t0
You edit a file inside the container — its writable layer.
t1
The edit is visible inside that container.
t2
You destroy the container (stop + rm).
t3
The edit is gone. The image is untouched — it was on the shelf this whole time.
t4
You start a new container from the same image: fresh, no marker.
Container edits do not reach the image. Ever. To change what the image contains, you change the recipe (the Dockerfile) and re-build. That is why, on a server, "I'll just tweak it live" is a trap — your tweak dies on the next deploy.
