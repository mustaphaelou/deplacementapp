Lesson 0001 · Docker mental model

Image vs container — and the two quieter nouns

Mission: deploy deplacementapp on Coolify · 5–7 min

You've run docker compose up and it "just worked" — until it didn't. The reason it stopped making sense is that four different nouns were all being called "Docker thing" in your head. Separate them now and the rest of the deployment, including what Coolify does on your behalf, will snap into place.

The four nouns, in one breath

Image
A read-only, layered filesystem snapshot plus some metadata (env vars, working directory, the command to run). Built from a Dockerfile. Stopped. Static. Boring.
Container
A running instance of an image. A container adds a writable layer on top of the read-only image, its own process tree, its own network, its own copy of the env. The same image can launch dozens of containers.
Layer
One instruction's worth of filesystem delta inside one image. Each RUN or COPY in a Dockerfile produces a layer. Layers stack and cache independently — so order matters.
Stage
One FROM block inside a multi-stage Dockerfile. Each stage builds its own intermediate image; later stages can COPY --from= bits out of earlier ones. Stages are the trick that keeps prod images small.
You don't run an image. You start an image — that act creates a container. The image is the recipe; the container is the cake. You can bake two cakes from one recipe. Tweaking a running container does not change the recipe.

See it in this repo

Open the repo's Dockerfile and count the FROM lines:

Dockerfile FROM node:20-alpine AS base FROM base AS deps FROM base AS prod-deps FROM base AS builder FROM base AS runner

That's five FROM lines = five stages (base is a shared parent; the other four are real build stages). Only the very last one — runner — becomes the image you ship. The first four are scaffolding: deps installs everything (for building), prod-deps installs only runtime deps, builder builds Next.js. Then runner COPY --from=es just the bits it needs from each.

Why this matters for you

When the deploy breaks, the symptom lives in the container (a process exiting, a port not answering, a healthcheck failing). But the fix almost always lives in the Dockerfile, which is the image recipe. If you can't tell image from container, you can't tell "I need to edit the Dockerfile" from "I need to restart the container" — and you'll waste hours trying the wrong one.

One quick self-check

All four phrases are the same length and structure — no clue from formatting. Pick before you peek at the answer. (This is retrieval practice; the slight discomfort is what builds storage strength.)

Your win for this lesson

You can now read the repo's Dockerfile and see five stages instead of five mysterious lines, and you know which one ends up actually running. That's the foundation everything else stands on.
