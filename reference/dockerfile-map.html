Companion to Lesson 0001 — Image vs Container.

The four nouns

TermWhat it ISWhat it is NOT
Image Read-only, layered filesystem snapshot + metadata (env, workdir, entrypoint). Built from a Dockerfile. Stopped. A running thing. A virtual machine. A package manager output.
Container A running instance of an image: isolated process tree, a writable layer stacked on the image, own network + env. Same image → many containers. The image itself. Always long-lived (containers can be one-shot).
Layer One instruction's worth of filesystem delta inside an image (one RUN or COPY). Stacked + cached independently. A build stage. A commit.
Stage One FROM block in a multi-stage Dockerfile. Produces an intermediate image; later stages COPY --from= out of it. A layer. A separate container.

The mental move

You don't run an image. You start an image; that act creates a container. The image is the recipe; the container is the cake. Two cakes from one recipe — identical. You can't change a recipe by tasting a cake: editing a running container does not change its image.

This repo's Dockerfile — the 4-stage pipeline

Stage 1 deps
  • full node_modules
  • npm ci --ignore-scripts
  • build-time only
Stage 2 prod-deps
  • node_modules w/o devDeps
  • npm ci --omit=dev
  • for runtime scripts
Stage 3 builder
  • copies deps ndm
  • prisma generate
  • npm run build → .next/standalone
Stage 4 · runner runner (final)
  • non-root nextjs user
  • standalone + static + public
  • prisma client + prisma CLI
  • HEALTHCHECK on /api/health

Only FROM … AS runner becomes the image you ship. Stages 1–3 are scaffolding to build it; they never reach the registry.

Why split stages?

ConcernHow the split handles it
Build tools in prod imagebuilder holds devDeps & source; runner takes only what the running app needs.
Image sizerunner doesn't include .next source — just the standalone server + static assets + a few prisma dirs.
Cache reuseChanging app code rebuilds builder + runner but reuses the cached deps & prod-deps stages (only triggered by package*.json change).
Runtime CLI accessprod-deps keeps prisma CLI available in runner for migrate deploy in the entrypoint.

Common gotchas (preview)

  • Editing package.json invalidates the deps+prod-deps+builder caches — expect a full reinstall next build.
  • npm ci --ignore-scripts skips postinstall prisma generate on purpose; the Dockerfile runs npx prisma generate explicitly in builder.
  • The runner user is 1001. Volumes mounted into public/uploads & pdfs must be writable by uid 1001, or the app silently fails at write time.
