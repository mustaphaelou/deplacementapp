Collapse the route-level triplication for DemandeDeplacement transitions

Strong in-process

app/api/demandes/[id]/approuver/route.ts
app/api/demandes/[id]/rejeter/route.ts
app/api/demandes/[id]/retirer/route.ts

Before

flowchart LR
  A[approuver] --> S[demandeService.executeAction approuver]
  B[rejeter] --> S2[demandeService.executeAction rejeter]
  C[retirer] --> S3[demandeService.executeAction retirer]
  style A fill:#f1f5f9
  style B fill:#f1f5f9
  style C fill:#f1f5f9

Three route files with near-identical auth, error handling, and service-call logic. Deleting any one barely moves complexity elsewhere.

After

flowchart LR
  subgraph deep["ApplyActionModule"]
    S[auth - guard - executeAction - response]
  end
  A[POST /demandes/:id] --> deep
  style deep fill:#1e293b,stroke:#0f172a,stroke-width:2px,color:#fff

Single route. Action parameter drives workflow. All before/after hooks internal to the module.

Problem

approuver, rejeter, and retirer routes are triplets: they authenticate, authorise, select a comment, validate, call the same executeAction, then serialise the same error classes. The interface of each route (its signature, error modes, auth boilerplate) is almost the entire file. Delete one and nothing about the system gets simpler, the remaining two still carry the pattern. This is a textbook shallow module with zero leverage.

Solution

Replace the three route files with a single POST /demandes/:id/action route that accepts an discriminated union { action: approuver or rejeter or retirer, payload: ... }. Push the role guard, the missing-comment check, and the error-response mapping into one module.

Benefits

  • Locality: auth guard, validation, and error mapping live in one file
  • Leverage: one test suite covers approve, reject, and withdraw
  • Interface shrinks from three route signatures to one
  • Deleting either the old approuver or rejeter route re-legitimises the complexity, confirms the triplication was the problem

Extract PDF rendering from route into a seam

Worth exploring local-substitutable

lib/pdf.ts
app/api/demandes/[id]/pdf/route.ts

Before

flowchart LR
  R[pdf/route.ts] --> P["lib/pdf.ts generatePdfBuffer()"] --> C["@react-pdf/renderer"] --> DB[(prisma.document.create)]
  style C fill:#fee2e2,stroke:#dc2626

Route mixes fetching data, calling React renderer, DB write, and HTTP response.

After

flowchart LR
  R[pdf/route.ts] --> M["PdfRenderModule interface: render(data)"]
  M --> A["Adapter: ReactPdfAdapter"]
  subgraph seam ["seam: PdfRenderModule"]
    M
  end
  style M fill:#1e293b,stroke:#0f172a,color:#fff

Route passes plain data; module picks adapter. Two adapters = real seam.

Problem

pdf/route.ts fetches a database row, calls generatePdfBuffer (which imports React and @react-pdf/renderer), writes a Document to the database, and streams a response. The route has no seam to swap the renderer, tests that hit this route are forced through React-PDF side effects. The module is shallow because its interface (fetch + render + persist + stream) is identical to its implementation.

Solution

Introduce PdfRenderModule with a thin interface: renderPdf(data, adapter) to Buffer. The route passes the DemandeDeplacement as data; the module delegates to an adapter. Provide a ReactPdfAdapter for production and a StubAdapter for tests. The database write can move into a separate side-effect hook or stay in the route, but the renderer itself becomes a seam.

Benefits

  • Locality: renderer concerns (fonts, layout, page breaks) are isolated
  • Leverage: stub adapter means fast, hermetic tests for the route
  • Two adapters make the seam real

Consolidate role dashboard data into a seam

Strong in-process

lib/dashboard.ts app/(dashboard)/page.tsx

Before

flowchart LR
  Page["(dashboard)/page.tsx"] --> getEmployeeDashboard
  Page --> getManagerDashboard
  Page --> getFinanceDashboard
  Page --> getDirectionDashboard
  style getEmployeeDashboard fill:#f1f5f9
  style getManagerDashboard fill:#f1f5f9
  style getFinanceDashboard fill:#f1f5f9
  style getDirectionDashboard fill:#f1f5f9

Four functions with duplicated filtering, ordering, Prisma include logic. No single interface for get dashboard for role.

After

flowchart LR
  subgraph deep ["DashboardQueryModule"]
    M["for(role, userId) to data"]
  end
  Page --> deep
  style deep fill:#1e293b,stroke:#0f172a,color:#fff

One module, one interface. Role resolves to a policy object internally.

Problem

lib/dashboard.ts has four functions: getEmployeeDashboard, getManagerDashboard, getFinanceDashboard, getDirectionDashboard. Each repeats the same Prisma query shape (filter by statut, include employee, order by timestamp, limit). The interface of each function is almost identical to its implementation, delete one and the others remain the same shape. No leverage.

Solution

Create DashboardQueryModule.for(role, userId) that returns the appropriate dashboard data. Internally, it uses a policy table (role to { statuts, orderBy, includeBudget }) to configure a single query builder. The page calls one function.

Benefits

  • Locality: dashboard query logic lives in one place
  • Leverage: add a role by adding one row in the policy table
  • Interface shrinks from four functions to one

Centralise auth guard into a single seam

Worth exploring mock

app/api/**/*.route.ts (approx 12 route files)

Before

flowchart LR
  R1[route1.ts] --> |duplicated| A1["auth() - 401 role check - 403"]
  R2[route2.ts] --> |duplicated| A2["auth() - 401 role check - 403"]
  R3[route3.ts] --> |duplicated| A3["auth() - 401 role check - 403"]

Every route repeats the same await auth() call, if (!session?.user) return 401, and role check.

After

flowchart LR
  R1[route1.ts] --> G["withAuth(role, handler)"]
  R2[route2.ts] --> G
  R3[route3.ts] --> G
  style G fill:#1e293b,stroke:#0f172a,color:#fff

withAuth is a single seam. Swapping its implementation (cookie vs JWT vs API key) touches one file.

Problem

Every API route repeats: const session = await auth(); if (!session?.user) return 401. Routes that need role checks add a second block. The auth guard is a shallow wrapper with no leverage, the caller (the route) already knows about NextRequest, NextResponse, and auth().

Solution

Introduce a withAuth(handler, requiredRole?) higher-order function that wraps the route handler. It handles the session check, the role check, and the 401/403 response. The route handler receives the already-validated session object.

Benefits

  • Locality: auth concern changes in one place
  • Leverage: every route handler shrinks
  • Two adapters (Next-Auth, test stub) make the seam real

Merge audit.ts into audit-bus.ts

Strong in-process

lib/audit.ts
lib/audit-bus.ts

Before

flowchart LR
  R1[utilisateurs/route.ts] --> audit["lib/audit.ts ajouterAudit()"]
  R2[demande-service.ts] --> bus["lib/audit-bus.ts AuditBus"]
  style audit fill:#fee2e2,stroke:#dc2626

Two modules do the same thing. One is imported by routes, the other by the service. Risk of drift.

After

flowchart LR
  R1[utilisateurs/route.ts] --> bus["AuditBus interface: log()"]
  R2[demande-service.ts] --> bus
  style bus fill:#1e293b,stroke:#0f172a,color:#fff

All callers use the same AuditBus seam. Adapter remains Prisma for now; can fake in tests.

Problem

lib/audit.ts exports ajouterAudit, a direct Prisma wrapper with no interface. lib/audit-bus.ts exports AuditBus, a proper module with an AuditAdapter seam. Both write to JournalAudit. The existence of audit.ts leaks the Prisma dependency into app/api/utilisateurs/route.ts (the route imports Prisma just to pass it to ajouterAudit). Deleting audit.ts would force the route to use the AuditBus seam instead, concentrating the logic.

Solution

Delete lib/audit.ts. Move callers to auditBus.log. The AuditBus module already has the seam (via AuditAdapter).

Benefits

  • Locality: audit logging logic in one module
  • Interface is the test surface for audit
  • Deletion test passes: removing audit.ts makes the codebase simpler

Top recommendation

Collapse the route-level transition triplication (Candidate 1)

It would remove the most surface area with the least risk. approuver, rejeter, and retirer routes are identical twins; merging them into a single POST /demandes/:id/action route would shrink the API surface by 66 percent, centralise auth/validation, and make the remaining interface the single place where transition actions are enumerated. This is also the fastest to validate, if it works, it confirms the pattern for the rest of the API.
